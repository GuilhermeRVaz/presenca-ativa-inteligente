# Contexto do Projeto: PAI (Presença Ativa Inteligente)
Sistema de automação de busca ativa escolar, orquestrando webhooks do WhatsApp para contactar responsáveis por alunos faltosos.

## Stack Tecnológica
- **Backend:** Python 3.13 com FastAPI.
- **Banco de Dados:** Supabase (PostgreSQL) usando a biblioteca oficial `supabase-py`.
- **Automação:** n8n e LangGraph para orquestração de IA e agentes triadores.
- **Mensageria:** Evolution API (WhatsApp).

## Padrões de Arquitetura e Código
- Siga estritamente os princípios de Clean Architecture implementados no projeto (pastas `domain`, `application`, `infrastructure`, `api`).
- Use sempre Tipagem Estrita (Type Hints) e `dataclasses` para os modelos de domínio.
- Utilize injenção de dependência e `Protocols` para definir interfaces entre camadas.
- Tratamento de exceções não deve vazar detalhes da infraestrutura para a API; logue erros e retorne respostas padronizadas.

## Regras de Negócio Críticas
1. **Resolução de Identidade (O Problema do LID):** A Evolution API frequentemente envia webhooks mascarando o número real do remetente com um "LID" (ex: `123456@lid`). O sistema NUNCA deve assumir que o remetente sempre terá um `wa_jid` válido no formato `5511...`.
2. **Triagem:** Qualquer identidade recebida como `UNRESOLVED` deve ser enviada para a fila de triagem da IA (n8n) para cruzar o contexto da conversa com a tabela `busca_ativa_v2.students`.
3. **Persistência:** O banco de dados utiliza o schema `busca_ativa_v2`. Todas as queries e RPCs no Supabase devem referenciar explicitamente este schema.

## Fluxo de Trabalho (Skills)
- Antes de modificar uma função de banco de dados, verifique o arquivo `repositories.py`.
- Sempre que criar uma rota nova no FastAPI, utilize `BackgroundTasks` para o processamento pesado de webhooks para garantir retorno HTTP 200 rápido para a Evolution API.
